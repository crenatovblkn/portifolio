Resumo Executivo

Durante uma investigação forense, o analista raramente encontra uma única evidência capaz de explicar todo um incidente de segurança.

Na maioria das situações, o trabalho consiste em reunir pequenos vestígios e estabelecer relações entre eles até reconstruir uma sequência lógica dos acontecimentos.

Neste laboratório será demonstrado como utilizar Bodyfile e mactime para gerar uma Timeline forense e identificar atividades realizadas em um sistema Linux.

Objetivo do laboratório

Demonstrar como metadados do sistema de arquivos podem ser utilizados para reconstruir eventos relacionados à criação de arquivos, persistência e possíveis atividades de exfiltração.

O desafio

Imagine que uma equipe de resposta a incidentes recebeu uma imagem forense após uma suspeita de comprometimento.

Não existem logs centralizados.

Não existe EDR.

Não existe antivírus capaz de explicar toda a atividade executada.

Existe apenas uma evidência digital.

E uma pergunta precisa ser respondida.

O que aconteceu?

Essa pergunta será respondida ao longo deste laboratório por meio da análise cronológica dos metadados presentes no sistema de arquivos.

Entendendo o Bodyfile

Para reconstruir uma sequência de eventos é necessário identificar os vestígios deixados pelo sistema operacional. Uma das formas mais utilizadas em investigação forense consiste na análise dos metadados armazenados pelo sistema de arquivos.

Esses metadados podem ser extraídos e organizados em um arquivo conhecido como Bodyfile, que posteriormente será utilizado para gerar uma Timeline cronológica das atividades executadas.

root@crportal:~# 

M = Modified

A = Accessed

C = Changed

B = Birth

Cada atributo representa uma informação importante sobre o ciclo de vida dos arquivos e diretórios presentes na evidência digital.

Observação do Analista

Um único timestamp raramente explica um incidente. O verdadeiro valor da análise forense está na correlação entre diversos eventos.

Preparando a Evidência

Para demonstrar o processo de investigação foi criado um pequeno laboratório controlado.

O primeiro passo consistiu na criação de uma imagem de disco de 200 MB, posteriormente formatada utilizando Ext4 e montada em um diretório específico.

analista@dfir:~$ 

$ dd if=/dev/zero of=evidencia.img bs=1M count=200

200+0 records in

200+0 records out

209715200 bytes copied

$ mkfs.ext4 evidencia.img

Creating filesystem...

done.

$ mkdir montagem

$ sudo mount evidencia.img montagem

$ ls

montagem/

A partir desse momento qualquer atividade realizada passaria a fazer parte da futura evidência digital.

O objetivo deste laboratório não é simular um ataque sofisticado. O objetivo é gerar artefatos suficientes para demonstrar como a Timeline Analysis pode ser utilizada durante uma investigação.

Uma breve pausa

Até este ponto ainda não existe um incidente. Existe apenas uma evidência preparada para registrar atividades.

Na prática, essa etapa é extremamente importante. Durante uma investigação real, a preservação da integridade da evidência determina a confiabilidade dos resultados obtidos pelo analista.

Importante

Toda alteração realizada após a aquisição da evidência pode comprometer a investigação. Preservar a integridade dos dados é um dos princípios fundamentais da Computação Forense.

Simulando um Incidente

Com a evidência preparada, chegou o momento de criar alguns artefatos que serão posteriormente identificados durante a investigação.

O objetivo não é reproduzir um ataque complexo. O objetivo é gerar rastros capazes de demonstrar como uma Timeline forense pode ser utilizada para reconstruir uma sequência de eventos.

Primeira ação: criando um Payload

O primeiro passo consiste na criação de um pequeno arquivo chamado payload.sh.

Após sua criação, um conteúdo simples é gravado no arquivo, simulando um possível script malicioso.

root@laboratorio:~# 

$ touch payload.sh

$ sleep 5

$ echo "malware" > payload.sh

$ ls

payload.sh
Observação do Analista

Os pequenos intervalos entre os comandos foram inseridos propositalmente para facilitar a interpretação da Timeline.

Segunda ação: Persistência

Ataques reais raramente executam apenas um payload. Frequentemente mecanismos de persistência são implantados para garantir acesso futuro ao sistema comprometido.

root@laboratorio:~# 

$ mkdir persistencia

$ sleep 5

$ touch persistencia/backdoor

$ ls persistencia/

backdoor

Neste laboratório, o diretório persistencia representa um mecanismo utilizado para manter acesso ao ambiente.

Terceira ação: Possível Exfiltração

Uma última atividade foi executada para simular o armazenamento temporário de informações antes de uma possível exfiltração.

root@laboratorio:/tmp# 

$ echo "dados roubados" > exfiltracao.txt

$ ls

exfiltracao.txt

Embora seja um laboratório simplificado, esse comportamento é compatível com situações encontradas em diversos incidentes reais.

Preservando a Evidência

Após a execução das atividades, a imagem forense foi desmontada.

Esse procedimento impede novas alterações e garante a integridade da evidência para análise posterior.

analista@dfir:~$ 

$ cd ..

$ sudo umount montagem

$ ls

evidencia.img
Importante

Uma investigação forense confiável depende da preservação da evidência. Qualquer alteração posterior pode comprometer os resultados obtidos durante a análise.

A Investigação Começa

Com a evidência preservada, inicia-se a fase de análise.

O primeiro passo consiste em extrair os metadados do sistema de arquivos.

analista@dfir:~$ 

$ fls -r -m / evidencia.img > bodyfile.txt

$ ls

bodyfile.txt

O Bodyfile reúne milhares de informações relacionadas aos arquivos e diretórios presentes na evidência.

Esses dados serão processados pelo mactime para gerar a Timeline forense.

analista@dfir:~$ 

$ mactime -b bodyfile.txt > timeline.txt

$ less timeline.txt

...

10:23:19 .a.. /payload.sh

10:23:24 m.c. /payload.sh

...
Perguntas do Investigador

Quem criou o payload?

Quando ele foi alterado?

Existe algum mecanismo de persistência?

Há indícios de exfiltração?

A Timeline contém essas respostas.
