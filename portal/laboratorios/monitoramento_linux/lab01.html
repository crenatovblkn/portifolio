🎯 Objetivo

Neste laboratório, iremos construir um pequeno ambiente de monitoramento Linux.

Ao longo das atividades, serão produzidos eventos controlados que permitirão observar como diferentes ferramentas registram informações sobre o sistema e sobre a rede.

Ao final, essas evidências serão utilizadas para compreender as ações realizadas durante a simulação.

🖥 Cenário

Imagine um servidor Linux utilizado para administração remota.

Os acessos são realizados através do SSH e diversas atividades administrativas são executadas diariamente.

Entre essas atividades, podemos destacar a criação de usuários e alterações em arquivos críticos do sistema operacional.

Mas uma pergunta surge naturalmente.

Como identificar e registrar essas atividades?

🛠 Preparando o Ambiente

Antes de iniciar o monitoramento, precisamos preparar o servidor Ubuntu.

Nosso ambiente será utilizado para acesso remoto e para geração dos eventos que serão analisados durante o laboratório.

O primeiro passo é verificar se os componentes necessários estão disponíveis.

🔑 Verificando o OpenSSH

O protocolo SSH será utilizado para acesso remoto ao servidor.

Vamos verificar o status do serviço.

Ubuntu Server 

ubuntu:~#systemctl status ssh

● ssh.service
Loaded: loaded
Active: active (running)

O estado active (running) indica que o serviço está operacional.

📄 Verificando o Syslog

Grande parte dos eventos produzidos pelos serviços Linux é registrada através do Syslog.

Vamos verificar se o serviço está ativo.

Ubuntu Server 
  
root@ubuntu:~#systemctl status rsyslog

● rsyslog.service
Loaded: loaded
Active: active (running)

Esse serviço será importante durante a investigação.

📋 Primeira Validação

Até este momento, nosso ambiente já possui os serviços básicos necessários para o laboratório.

✅ SSH Ativo
✅ Syslog Ativo
🛠 Ubuntu Preparado

💡 Próxima Etapa

Com os sensores instalados e validados, iremos configurar as regras de monitoramento que serão utilizadas durante o laboratório.

⚙ Configurando o Auditd

Com os sensores instalados, vamos definir quais atividades serão monitoradas.

Neste laboratório, o objetivo será acompanhar alterações em um dos arquivos mais importantes do Linux.

/etc/passwd

Esse arquivo contém informações sobre as contas de usuários do sistema operacional.

🖥 Criando a Regra

Vamos adicionar uma regra simples ao Auditd.

Ubuntu Server 

root@ubuntu:~#nano /etc/audit/rules.d/audit.rules
-w /etc/passwd -p wa -k passwd_changes

🔄 Aplicando a Configuração

Após criar a regra, precisamos carregá-la.

Ubuntu Server 

root@ubuntu:~#augenrules --load
root@ubuntu:~#systemctl restart auditd

✅ Validando o Auditd

Vamos verificar se a regra foi carregada.

Ubuntu Server 

root@ubuntu:~#auditctl -l
-w /etc/passwd -p wa -k passwd_changes

A presença da regra confirma que o monitoramento está ativo.

👁 Configurando o Snort

Também iremos adicionar uma regra simples para identificar conexões SSH.

Ubuntu Server 

root@ubuntu:~#nano /etc/snort/rules/local.rules

alert tcp any any -> any 22
(msg:"LAB SSH CONNECTION";
sid:1000001;
rev:1;)

✅ Validando o Snort

Antes de continuar, vamos verificar a configuração.

Ubuntu Server 

root@ubuntu:~#snort -T -c /etc/snort/snort.conf

Snort successfully validated
the configuration.

Snort exiting


A validação indica que a configuração não possui erros.

Neste momento, nosso ambiente já possui sensores configurados e prontos para utilização.

✅ Auditd Configurado
✅ Snort Configurado

O objetivo é verificar como atividades administrativas podem gerar registros nos mecanismos de monitoramento.

🔑 Estabelecendo uma Conexão SSH

No Kali Linux, vamos iniciar uma conexão com o servidor Ubuntu.

Kali linux 

kali@kali:~$ssh aluno@192.168.1.10
aluno@192.168.1.10's password:
Welcome to Ubuntu
Last login:
aluno@ubuntu:~$


A conexão foi estabelecida com sucesso.

👤 Criando um Novo Usuário

Após acessar o servidor, vamos criar um novo usuário.

Kali linux 

aluno@ubuntu:~$sudo useradd laboratorio


Essa operação provocará alterações em arquivos do sistema operacional.

🔒 Definindo uma Senha

Vamos configurar uma senha para o usuário criado.

Kali linux 

aluno@ubuntu:~$sudo passwd laboratorio
New password:
Retype new password:
passwd:password updated successfully

A operação foi concluída com sucesso.

🖥 Consultando o Auditd

🔎 Consultando o ausearch

Também podemos realizar consultas específicas aos eventos registrados.

Ubuntu Server 

root@ubuntu:~#ausearch -k passwd_changes
----

type=SYSCALL
comm="useradd"
name="/etc/passwd"
key="passwd_changes"


A pesquisa retornou os eventos relacionados à regra criada.

🌐 Investigando a Rede

Já confirmamos que o sistema operacional registrou as atividades realizadas durante o laboratório.

Agora precisamos verificar se a infraestrutura de monitoramento da rede também observou esses eventos.

👁 Consultando o Snort

Durante a configuração, criamos uma regra para detectar conexões SSH.

Vamos consultar os alertas produzidos.

Ubuntu Server 

root@ubuntu:~#tail -f /var/log/snort/alert

[**]
[1:1000001:1]
LAB SSH CONNECTION
[**]
TCP
192.168.1.20:52144
->
192.168.1.10:22

🔍 Interpretando o Alerta

O alerta fornece informações importantes sobre a conexão.

Origem 192.168.1.20
Destino 192.168.1.10
Serviço SSH
Porta 22

Essas informações permitem identificar a comunicação realizada durante o laboratório.

🖥 O que a Rede Registrou?

Até este momento, a investigação da rede permitiu confirmar que uma conexão SSH foi observada.

Essa informação, isoladamente, não caracteriza uma atividade maliciosa.

Entretanto, ela representa uma importante fonte de evidência que poderá ser comparada com os registros do sistema operacional.
