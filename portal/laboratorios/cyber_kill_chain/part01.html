🎯 O Cenário

Imagine um servidor Linux disponibilizado para acesso remoto.

Administradores e usuários autorizados utilizam o protocolo SSH para realizar suas atividades.

Entretanto, um dos desafios das equipes de segurança é identificar quando essas conexões podem representar um comportamento suspeito.

Além disso, alterações críticas no sistema operacional, como a criação de novos usuários, também precisam ser monitoradas.

Neste laboratório, iremos construir um pequeno ambiente capaz de produzir evidências úteis para investigação.

💡 Objetivo do Laboratório

Este laboratório não tem como objetivo simular um ataque real. Nosso foco será compreender como diferentes mecanismos de monitoramento podem registrar eventos importantes do sistema operacional e da rede.

🎯 O que iremos observar?

  • ✅ Conexões SSH.
  • ✅ Alterações no arquivo /etc/passwd.
  • ✅ Criação de usuários.
  • ✅ Eventos produzidos pelo Auditd.
  • ✅ Alertas produzidos pelo Snort.
  • ✅ Registros do Syslog.
  • ✅ Primeiras evidências para investigação.

🛠 Preparando o Servidor

Nosso primeiro passo será preparar o servidor Ubuntu para receber conexões SSH.

Além disso, iremos verificar se os principais serviços necessários ao laboratório estão funcionando corretamente.

🔑 Verificando o OpenSSH

O protocolo SSH será utilizado para acesso remoto ao servidor.

Vamos verificar se o serviço está instalado e em execução. 


root@ubuntu:~#

systemctl status ssh

● ssh.service

Active: active (running)

Loaded: loaded

Main PID: 852

O estado active (running) indica que o serviço está disponível.

💡 O Analista Observa

Antes de qualquer atividade de monitoramento, é importante validar os serviços que serão utilizados no laboratório. Problemas nesta etapa podem gerar resultados inconsistentes durante a investigação.

⚙ Configurando o Auditd

Com os serviços instalados, vamos configurar o monitoramento de alterações críticas no sistema operacional.

Nosso foco será o arquivo

/etc/passwd

responsável pelas informações das contas de usuários locais.

🖥 Criando a Regra

Adicione a seguinte regra ao Auditd. 


root@ubuntu:~#

nano /etc/audit/rules.d/audit.rules

-w /etc/passwd -p wa -k passwd_changes

Essa configuração permitirá registrar alterações no arquivo.

🔍 Entendendo a regra

-w Monitora o arquivo.

-p wa Registra operações de escrita e alteração de atributos.

-k passwd_changes Cria uma chave para facilitar consultas futuras.

🔄 Carregando as regras 


root@ubuntu:~#

augenrules --load

root@ubuntu:~#

systemctl restart auditd

Após carregar as regras, o monitoramento estará ativo.

✅ Validando o Auditd 


root@ubuntu:~#

auditctl -l

-w /etc/passwd -p wa -k passwd_changes

A presença da regra confirma que o monitoramento foi habilitado.

👁 Criando a Regra do Snort

Também iremos criar uma regra simples para detectar conexões SSH. 


root@ubuntu:~#

nano /etc/snort/rules/local.rules

alert tcp any any -> any 22

(msg:"LAB SSH CONNECTION";

sid:1000001;

rev:1;)
💡 Entendendo a Regra

alert Gera um alerta.

tcp Monitora tráfego TCP.

22 Porta padrão do SSH.

msg Mensagem exibida no alerta.

sid Identificador único da regra.

rev Versão da regra.

🔄 Validando a Configuração

Antes de iniciar a simulação, é importante verificar a sintaxe do Snort. 


root@ubuntu:~#

snort -T -c /etc/snort/snort.conf

Snort successfully validated
the configuration.

Snort exiting

A validação garante que a configuração não possui erros.

👨‍💻 O Analista Observa

Um erro bastante comum durante implantações é assumir que uma configuração foi aplicada sem realizar testes. A validação das regras antes da geração dos eventos reduz significativamente problemas futuros durante investigações.

🕵 Primeira Pergunta

O sistema operacional registrou alterações relacionadas ao laboratório?

🖥 Consultando o Auditd

Uma forma simples de acompanhar os eventos é consultar o arquivo de auditoria. 


root@ubuntu:~#

tail -f /var/log/audit/audit.log

type=SYSCALL

type=PATH

name="/etc/passwd"

key="passwd_changes"

comm="useradd"

O evento confirma que alterações foram registradas.

💡 Interpretando o Evento

name="/etc/passwd" Arquivo monitorado.

comm="useradd" Processo executado.

key="passwd_changes" Regra criada durante o laboratório.

🔎 Utilizando o ausearch

O Auditd permite realizar consultas específicas aos eventos registrados. 


root@ubuntu:~#

ausearch -k passwd_changes

----

time->

type=SYSCALL

comm="useradd"

name="/etc/passwd"

key="passwd_changes"

A pesquisa retornou os eventos relacionados à regra criada.

👨‍💻 O Analista Observa

O uso de chaves durante a criação das regras facilita a localização dos eventos durante uma investigação.
