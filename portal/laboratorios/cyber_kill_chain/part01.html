🎯 O Cenário

Imagine que uma organização possui um servidor Linux publicado para acesso remoto.

Um usuário estabelece uma conexão SSH com esse servidor.

Após obter acesso, novas contas são criadas e arquivos críticos do sistema operacional são modificados.

Uma pergunta surge naturalmente.

Como identificar que essas ações estão acontecendo?

Neste laboratório, vamos construir um ambiente simples capaz de detectar eventos de rede e alterações em arquivos críticos do sistema Linux.

💡 Observação do Analista

Este laboratório não pretende reproduzir um ataque real. Seu objetivo é demonstrar como diferentes mecanismos de segurança podem gerar evidências úteis para investigação e resposta a incidentes.

🎯 Objetivos

Ao concluir este laboratório, seremos capazes de observar:

  • ✅ Tentativas de acesso SSH.
  • ✅ Monitoramento do arquivo /etc/passwd.
  • ✅ Criação de usuários locais.
  • ✅ Eventos produzidos pelo Auditd.
  • ✅ Alertas produzidos pelo Snort.
  • ✅ Correlação entre eventos.
  • ✅ Evidências para investigação.

⚙ Configurando o Auditd

Agora que o Auditd está instalado, precisamos informar quais eventos serão monitorados.

Neste laboratório, nosso foco será acompanhar alterações em um dos arquivos mais importantes do Linux.

/etc/passwd

Esse arquivo contém informações relacionadas às contas de usuários do sistema operacional.

💡 Por que monitorar o /etc/passwd?

A criação de novos usuários ou alterações indevidas podem indicar tentativas de persistência após um comprometimento do ambiente. Por esse motivo, esse arquivo costuma receber atenção especial durante investigações forenses.

🖥 Criando a regra

Vamos adicionar uma regra simples ao Auditd. 


root@ubuntu:~# nano /etc/audit/rules.d/audit.rules

-w /etc/passwd -p wa -k passwd_changes

Essa configuração informa ao Auditd que alterações nesse arquivo devem ser registradas.

🔍 Entendendo a regra

-w Monitora um arquivo.

-p wa Observa operações de escrita e alteração de atributos.

-k passwd_changes Cria uma chave que facilitará pesquisas futuras durante a investigação.

🔄 Recarregando as regras

Após realizar a configuração, precisamos informar ao Auditd que novas regras foram adicionadas. 


root@ubuntu:~# augenrules --load

No change

root@ubuntu:~# systemctl restart auditd

A partir deste momento, o monitoramento está ativo.

✅ Validando a configuração

Antes de iniciar a simulação, é importante verificar se a regra foi carregada. 


root@ubuntu:~# auditctl -l

-w /etc/passwd -p wa -k passwd_changes

A presença da regra indica que qualquer modificação futura no arquivo será registrada.

👨‍💻 O Analista Observa

Uma prática comum em equipes Blue Team é validar todas as configurações antes de iniciar a coleta de evidências. Detectar problemas nesta etapa é muito mais simples do que descobrir, durante uma investigação, que eventos importantes não foram registrados.

📚 Relação com a Cyber Kill Chain

Nesta etapa, iremos observar algumas fases de uma cadeia de ataque simplificada. Inicial Access. Execution. Persistence. Cada ação produzirá evidências importantes para investigação.

🔑 Primeira ação Estabelecendo acesso

Nosso atacante inicia uma conexão SSH com o servidor Ubuntu. 


kali@attacker:~$ ssh aluno@192.168.1.10

aluno@192.168.1.10's password:

Welcome to Ubuntu

Last login:

aluno@ubuntu:~$

Do ponto de vista do sistema operacional, essa ação é legítima.

Entretanto, dependendo do contexto, ela pode representar o início de uma atividade maliciosa.

💡 O Analista Observa

Uma conexão SSH, por si só, não representa um incidente. Entretanto, durante uma investigação, o horário, a origem e o contexto da conexão podem fornecer informações importantes.

⚙ Segunda ação Criando persistência

Após obter acesso, o atacante decide criar um novo usuário.

Essa técnica é bastante conhecida e pode permitir acessos futuros ao ambiente. 


aluno@ubuntu:~$ sudo useradd invasor

aluno@ubuntu:~$ sudo passwd invasor

New password:

Retype new password:

passwd:

password updated successfully

Essa ação provocará uma alteração no arquivo

/etc/passwd

que está sendo monitorado pelo Auditd.

⚠ Simulação Controlada

Neste laboratório, a criação do usuário serve apenas para gerar evidências. Em um incidente real, essa atividade poderia indicar uma tentativa de persistência do atacante.

🎯 O que esperamos?

Até este momento, duas atividades importantes ocorreram.

Essas ações deverão produzir evidências que poderão ser observadas pelos mecanismos de segurança do laboratório.

🕵 Perguntas do Investigador

Uma investigação normalmente começa com algumas perguntas. Quem realizou a ação? Quando aconteceu? Qual processo foi utilizado? Qual arquivo foi afetado? Existe alguma relação entre os eventos?

🖥 Primeira Evidência Auditd

Como configuramos o monitoramento do arquivo

/etc/passwd

esperamos encontrar um evento relacionado à criação do usuário. 


root@ubuntu:~#

tail -f /var/log/audit/audit.log

type=SYSCALL

type=PATH

name="/etc/passwd"

key="passwd_changes"

auid=1000

uid=0

comm="useradd"
🔍 Interpretando a Evidência

name="/etc/passwd" Identifica o arquivo alterado.

comm="useradd" Mostra o processo responsável.

uid=0 Indica privilégios administrativos.

auid=1000 Permite identificar o usuário que iniciou a sessão.

👨‍💻 O Analista Observa

Perceba que o Auditd não apenas informa que um arquivo foi alterado.

Ele fornece informações importantes para reconstruir o incidente.

  • ✅ Qual arquivo.
  • ✅ Qual processo.
  • ✅ Quem executou.
  • ✅ Com quais privilégios.

Esses dados serão fundamentais para a correlação dos eventos.

💡 O papel do Snort

Enquanto o Auditd observa atividades realizadas no sistema operacional, o Snort monitora eventos de rede. Neste laboratório, utilizaremos uma regra simples para identificar tentativas de conexão SSH.

🖥 Consultando os alertas

Após a conexão SSH, esperamos encontrar um evento correspondente. 


root@ubuntu:~#

tail -f /var/log/snort/alert

[**]

[1:1000001:1]

LAB SSH CONNECTION DETECTED

[**]

TCP

192.168.1.20:54321

->

192.168.1.10:22

Nosso IDS registrou a tentativa de conexão.

🔍 Interpretando a Evidência

192.168.1.20 Representa o possível atacante.

192.168.1.10 Servidor monitorado.

Porta 22 Serviço SSH.

LAB SSH CONNECTION DETECTED Mensagem produzida pela regra criada durante o laboratório.
